This repository contains the scripts and topology code used to run the QoE forecasting experiments. The project mixes Mininet/Containernet topologies, Docker-based edge and cloud services, and shell wrappers that automate the different experiment scenarios.

## What Is Included

- `scripts/` contains the main shell entrypoints for reproducing the experiments.
- `topology/` contains the Mininet/Containernet simulation drivers.
- `edge/`, `cloud/`, `player/`, and `services/` provide the runtime components used by the experiments.

## Requirements

The experiments depend on a Linux environment with:

- Docker
- Python 3
- Mininet / Containernet
- Wi-Fi and wireless simulation support used by the topology code
- SSH access for the scripts that run on remote hosts

Python dependencies are split by component:

- `player/requirements.txt`
- `edge/requirements.txt`
- `services/steering/requirements.txt`

The topology scripts also expect the container images `cloud/apache`, `dind`, and `eduardogama/edge_apache` to be available locally or pullable from your registry.

## Setup

1. Install the Python dependencies for the components you plan to run.
2. Make sure Docker can run privileged containers, since the edge nodes are started as Docker containers inside the topology.
3. Verify the local hostnames and remote addresses used by the scripts. Several scripts contain hard-coded SSH users, keys, and IPs that you will likely need to change for your environment.
4. If you use the wireless topologies, ensure the host entries referenced in the topology files are present in `/etc/hosts`.

## Experiment Scripts

Run the scripts from the repository root.

### Docker timing scripts

These scripts benchmark image pulls and container startup time:

- `bash scripts/pull_docker_image.sh`
	- Repeatedly pulls `eduardogama/edge_apache` and records the time in `docker_pull_times_v2.log`.
- `bash scripts/deploy_docker_container.sh`
	- Repeatedly starts and stops a container from `eduardogama/edge_apache:latest` and records the time in `docker_deploy_times.log`.

### Topology and experiment runners

- `bash scripts/run_topo.sh`
	- Runs the main topology experiment through `topology/topo.py`.
	- Starts a node monitor in the background and writes logs under `logs/`.
	- The script uses the command `python topology/topo.py 10 20 2.0 $k $k` inside a loop, so confirm that the topology driver and its inputs match your local setup.

- `bash scripts/run_multi_edge_cloud_experiment.sh`
	- Runs the multi-edge cloud scenario with planner and monitoring processes.
	- Calls `python topology/run-multi-edge-cloud.py --users=... --arr_rate=... --seed=...`.
	- Aggregates output into directories named like `logs-multi-edge-cloud-10min-40ms-30u-dyn-k5-load-plan-reactive-*`.

- `bash scripts/run_one_edge_experiment.sh`
	- Runs the single-edge scenario.
	- Uses SSH to start the steering service on a remote host, then calls `python topology/run-one-edge.py --users=... --arr_rate=... --seed=...`.

- `bash scripts/run_edge_experiment.sh`
	- Runs the edge-cloud scenario.
	- Uses `python topology/run-edge-cloud.py --users=... --arr_rate=... --seed=...`.

- `bash scripts/run_cloud_experiment.sh`
	- Runs the cloud-only scenario.
	- Uses SSH to start the steering service and container monitor on a remote machine, then calls `python topology/run-cloud-only.py --users=... --arr_rate=... --seed=...`.

## Generated Logs And Dataset

The experiments produce several artifacts that are used later for analysis and model training:

- `scripts/docker_pull_times.log` and `scripts/docker_deploy_times.log`
	- Generated by the Docker timing scripts in `scripts/`.
	- They capture repeated pull and container-start timings for the Docker image benchmarks.
- `logs/<scenario>-<seed>/...`
	- Generated by the topology runners.
	- These folders typically contain per-run outputs such as `node_stats.csv`, `node_stats.db`, `aggregated_result.csv`, and other scenario-specific files.
- `services/planner/models/dataset.csv`
	- Generated by `services/planner/offline/trainning.py` after combining and filtering the collected run logs.
	- This is the dataset used for planner training and analysis.

The dataset-building helpers in [services/planner/common/dataset_utils.py](services/planner/common/dataset_utils.py#L1) load and transform these logs into the final training set. If you are regenerating the dataset, make sure the source `logs/` hierarchy matches the paths expected by the planner code.

## Notes About The Scripts

- Several scripts assume specific SSH identities, hostnames, and absolute paths such as `/root/.ssh/id_rsa_grifo_root` or `/home/eduardo/.ssh/id_rsa_dionisio`.
- Some wrappers reference experiment drivers that are not part of this repository snapshot, such as `topology/run-one-edge.py`, `topology/run-edge-cloud.py`, and `topology/run-cloud-only.py`. If you are reproducing the experiments, verify those files exist in your checkout or replace the calls with the correct local drivers.
- Most runs create log folders named `logs/<seed>` or scenario-specific archive directories. Review those outputs after each run before starting the next experiment.
- A few scripts end with `shutdown -h now`. Remove that line if you do not want the machine to power off automatically after the final run.

## Repository Layout

- `topology/` contains the simulation and experiment orchestration code.
- `services/` contains the steering and planner services used by the adaptive experiments.
- `edge/` and `cloud/` contain the containerized server-side components.
- `player/` contains the client-side playback assets and dependencies.

## Suggested Workflow

1. Pull or build the Docker images used by the experiments.
2. Install the Python requirements for the components you need.
3. Update the hard-coded SSH keys, hostnames, and filesystem paths in the shell scripts.
4. Run the smallest scenario first, usually `scripts/run_topo.sh`, to verify the environment.
5. Run the edge, cloud, and multi-edge experiments once the topology boots cleanly.
